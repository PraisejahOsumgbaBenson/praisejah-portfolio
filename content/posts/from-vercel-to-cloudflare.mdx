# My Next.js Hosting Journey: Netlify -> Vercel -> Cloudflare (and Why `fs` Broke My Blog)

## Intro

I built a Next.js blog with a simple goal: keep writing easy and the site fast.

At first, hosting didn't feel like a big decision. I just needed somewhere to deploy.

That changed as soon as I moved the project from Netlify to Vercel and later to Cloudflare Workers.

Each move worked until it didn't.

The moment things broke was small but important:

> `fs` stopped working.

And that's when it became clear:

> Where your code runs matters just as much as what your code does.

## The Original Setup

The blog started with a simple structure:

- Markdown files inside `/posts`
- Each post read and rendered dynamically
- File access handled with `fs`

It looked like this:

```ts
import fs from "fs";
import path from "path";

export function getPosts() {
  const postsDir = path.join(process.cwd(), "posts");
  const files = fs.readdirSync(postsDir);

  return files.map((file) => {
    const content = fs.readFileSync(path.join(postsDir, file), "utf-8");
    return {
      slug: file.replace(".md", ""),
      content,
    };
  });
}
```

It was simple, predictable, and worked exactly how I expected locally and during builds.

## Phase 1: Starting on Netlify

Netlify handled the first deployment without issues.

**What stood out:**

- Fast setup
- Smooth deployments
- Good default performance

But over time, the limitations started to show:

- Less control over how things run
- Some friction around server-side behavior

That's when I started looking for something more aligned with Next.js.

## Phase 2: Moving to Vercel

Moving to Vercel felt like the natural step.

Everything just worked:

- Routing
- Rendering
- Deployment

It felt like the environment Next.js was designed for.

But even though the experience improved, I still wanted:

- More control
- Better global performance

That's what led me to Cloudflare.

## Phase 3: Moving to Cloudflare Workers

Cloudflare Workers was appealing for one reason:

> Run code closer to users, everywhere.

The shift was immediate:

- Faster edge delivery
- Different execution model
- No traditional server

And then the issue showed up:

> `fs` is not defined

## The `fs` Problem

The problem wasn't obvious at first, because everything worked before.

Here's what was really happening:

- Locally -> full Node.js -> `fs` works
- Build step -> files exist -> `fs` works
- Edge runtime -> no filesystem -> `fs` fails

So this line:

```ts
fs.readFileSync(...)
```

was depending on something that simply doesn't exist in Cloudflare's runtime.

The issue wasn't the code itself. It was when the code was running.

## How I Fixed It

Instead of reading files at runtime, I moved that logic to build time.

### Step 1: Generate the data

```ts
// scripts/generatePosts.ts

import fs from "fs";
import path from "path";

const postsDir = path.join(process.cwd(), "posts");
const files = fs.readdirSync(postsDir);

const posts = files.map((file) => {
  const content = fs.readFileSync(path.join(postsDir, file), "utf-8");
  return {
    slug: file.replace(".md", ""),
    content,
  };
});

const output = `export const posts = ${JSON.stringify(posts)};`;

fs.writeFileSync("./lib/posts.data.ts", output);
```

### Step 2: Use static data at runtime

```ts
import { posts } from "@/lib/posts.data";

export function getPosts() {
  return posts;
}
```

### Step 3: Add it to the build process

```json
{
  "scripts": {
    "generate-posts": "ts-node scripts/generatePosts.ts",
    "build": "npm run generate-posts && next build"
  }
}
```

Now:

- No filesystem is needed at runtime
- The app works on Cloudflare Workers
- Everything is predictable

The only tradeoff is rebuilding when content changes, which is fine for a blog.

## The Bigger Lesson: Edge vs Traditional Hosting

This wasn't just about fixing `fs`.

It changed how I think about hosting:

- Traditional servers -> full control, fewer constraints
- Edge platforms -> faster globally, but stricter environments

The real tradeoff is simple:

> Flexibility vs distribution

## Cloudflare vs Vercel

Both are strong, just different.

- **Vercel** -> easiest way to run Next.js
- **Cloudflare** -> better for global performance

Vercel feels seamless.
Cloudflare feels powerful, but expects you to design for it.

## What I'd Do Differently

If I started again:

- I'd decide the deployment target earlier
- I'd avoid relying on runtime filesystem access
- I'd separate content processing from rendering from the start

## Who Should Use What

- **VPS** -> if you need full backend control
- **Vercel** -> if you want speed and simplicity with Next.js
- **Cloudflare Workers** -> if you want edge performance and are okay with constraints

## Conclusion

Moving across these platforms wasn't just about deployment. It changed how I design applications.

Each environment forced different decisions, and those decisions made the app better.

The one thing I'd keep in mind going forward:

> Build for where your code runs, not just where you write it.
